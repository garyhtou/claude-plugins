# Live Site Inspection (ground truth)

Direct browser inspection of real high-fidelity marketing sites, done with Claude
in Chrome on 2026-06-19. This is first-hand evidence (DOM, computed styles, WebGL
contexts, design tokens) to complement the web research, which relies on published
writeups.

> Note: stripe.com could not be inspected directly in the browser, so its findings
> come from published Stripe writeups (covered in the other research docs). The
> live inspection below is of column.com.

## column.com (inspected 2026-06-19)

**Build / stack**
- Generated by **Gatsby 5.16.1** (React-based static-site generator). Bundled
  with webpack (`webpack-runtime`, `framework`, `app` chunks). Libraries are
  module-scoped, so no animation globals (`gsap`, `THREE`, `Lenis`) are exposed
  on `window`. Their presence has to be inferred from behavior, not globals.
- React-based, statically pre-rendered (good for SEO + first paint on a
  marketing site), then hydrated.

**Typography**
- Primary typeface: **SuisseIntl** (a licensed premium grotesque), with a long
  system-font fallback stack. Using a paid, characterful typeface (rather than a
  default system or generic Google font) is itself a high-fidelity signal.
- `h1` rendered at 52px at desktop width. Type is treated as a first-class
  design element.

**Design tokens (the backbone)**
The single most reusable lesson. `:root` carries a deep, systematic set of CSS
custom properties. Observed samples:
- **Spacing scale** in rem with a consistent numeric ramp and NEGATIVE tokens
  for overlap/pull effects: `--space-7: 1.75rem`, `--space-16: 4rem`,
  `--space-36: 9rem`, `--space-40: 10rem`, `--space-n2: -0.5rem`,
  `--space-n10: -2.5rem`, `--space-n80: -20rem`, `--space-auto: auto`.
- **Color system** with numbered shades per hue (50 to 900) AND parallel `--rgb-*`
  triplet variants so colors can be composed with arbitrary alpha:
  `--color-purple-300: rgba(193,170,241,1)` alongside
  `--rgb-purple-900: 22,6,48`. The rgb-triplet pattern is a pro move: it lets you
  write `rgba(var(--rgb-purple-900), 0.4)` anywhere.
- **Typographic line-height tokens** split by role: body vs heading, e.g.
  `--lh-b-300: 1.375rem` (body), `--lh-h-1000: 4.5rem` (heading),
  `--lh-h-100: 0.875rem`.
- **Type sizes**: `--text-500: 1.5rem`, etc.
- **Grid templates** as tokens: `--cols-10: repeat(10, minmax(0, 1fr))`.
- **Semantic shadow tokens**: `--shadow-product: 0px 40px 32px rgba(0,0,0,0.02)`
  (note the very low alpha: premium shadows are soft and subtle, not heavy).
- **Semantic aliases** layered over primitives:
  `--bg-color-muted`, `--text-color-duotone-secondary`.

Takeaway: high-fidelity sites are built on a **two-tier token system** (raw
primitives -> semantic aliases). The skill should make establishing this the
first build step.

**Near-3D / WebGL**
- Two `<canvas>` elements with live **WebGL 2.0** contexts (confirmed via
  `getContext('webgl2')` returning a context; renderer string "WebKit WebGL"
  over ANGLE Metal on the test machine).
- Backing store is scaled to ~4x device-pixel-ratio for retina crispness: an
  800x800 CSS canvas has a 3200x3200 drawing buffer. (Rule: set canvas
  `width/height` to `cssSize * devicePixelRatio` and cap DPR at ~2 for perf.)
- The hero canvas is `pointer-events: none` and positioned as a decorative
  layer. The "near-3D" feel comes from shader-driven gradient/visual layers
  sitting behind/around real DOM content, not from making the whole page 3D.

**Motion / scroll approach**
- **Native scroll**, not scroll-hijacking: `body` uses `overflow: hidden auto`
  (native vertical scroll), and there is NO tall transform-translated wrapper
  (the signature of Lenis/Locomotive transform-based smooth scroll). They animate
  ON native scroll position rather than virtualizing the scroll.
- Heavy GPU compositing: **467 elements** carry a non-default `will-change`.
  This is a double-edged teaching point: `will-change` promotes elements to their
  own compositor layer (smooth transform/opacity animation), but applying it to
  hundreds of elements permanently wastes GPU memory. Best practice is to add it
  just before an animation and remove it after, or scope it narrowly. Seeing a
  top-tier site lean this hard on it shows the tradeoff is real, not a reason to
  copy the count.
- `scroll-behavior: auto` at the root (smoothness handled by their own JS /
  scroll-triggered reveals, not CSS smooth scroll).

## Cross-site patterns worth encoding in the skill

1. **Tokens first.** A real two-tier design-token system (primitives + semantic
   aliases) in CSS variables is the foundation, including negative spacing and
   rgb-triplet colors for alpha control.
2. **Premium type.** A licensed/characterful typeface, large editorial heading
   sizes, role-split line-height tokens.
3. **Decorative WebGL, not WebGL-everything.** Shader/gradient canvases as
   pointer-events-none layers behind real DOM give the "near-3D" wow without
   rebuilding the page in 3D. Retina-scale the backing store, cap DPR.
4. **Animate on native scroll** by default; reserve scroll-hijacking (Lenis)
   for when you specifically want eased/virtualized scroll. Native scroll keeps
   accessibility and performance simpler.
5. **Mind `will-change`.** Use it deliberately and transiently, not as a blanket.

## Method notes (for reproducibility)
Inspection ran JavaScript in the page (via Claude in Chrome, but any browser's
devtools console works the same way) to read `document.scripts`,
`getComputedStyle`, `:root` custom properties, and `canvas.getContext('webgl2')`.
To reproduce on any site: probe script srcs and
`meta[name=generator]` for the stack, enumerate `--*` custom properties on
`:root` for the token system, and query `canvas` elements + their WebGL context
for near-3D usage. Note many sites bundle libraries (no window globals), so infer
from behavior and DOM signatures rather than expecting `window.gsap` etc.
